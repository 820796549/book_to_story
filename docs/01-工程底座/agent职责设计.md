# agent 职责设计

## 1. 设计目标

agent 的作用不是制造复杂度，而是把拆书写文章过程中的不同判断分开。

本项目的 agent 只服务一个目标：

```text
稳定产出高质量拆书文章。
```

每个 agent 必须有清楚边界。

不能出现一个 agent 同时负责定位、反推、拆原文、写正文、审查所有事情。

## 2. agent 总表

| Agent | 职责 | 不负责 |
|---|---|---|
| story-architect | 书籍定位、目标用户、栏目体系 | 不写正文 |
| story-deslop | 样文反推、写法库沉淀 | 不判断原文真伪 |
| chapter-extractor | 原文拆解、证据提取 | 不写成品文章 |
| story-long-analyze | 模型提炼、推理链整理 | 不做情绪化表达 |
| story-long-analyze | 现实迁移、用户场景映射 | 不脱离原文发挥 |
| narrative-writer | 栏目文章成稿 | 不跳过前置材料 |
| style-polisher | 表达润色、节奏优化 | 不改核心论点 |
| consistency-checker | 质量审查、跑偏检测 | 不替代成稿写作 |
| project-packager | 项目打包、交付整理 | 不重新定义栏目 |

## 3. story-architect

### 3.1 核心职责

定义一本书为什么值得拆、给谁读、写成什么栏目。

### 3.2 输入

- 书名。
- 作者。
- 原文类型。
- 用户目标。
- 样文情况。
- 竞品情况。

### 3.3 输出

```text
00-项目定义/书籍定位.md
00-项目定义/目标用户.md
00-项目定义/栏目体系.md
00-项目定义/使用路径.md
00-项目定义/内容地图.md
```

### 3.4 禁止

- 不得直接写正文。
- 不得默认套用《毛选》栏目。
- 不得把“所有人”当目标用户。

## 4. story-deslop

### 4.1 核心职责

把竞品或样文拆成可学习写法。

### 4.2 输入

- 样文。
- 对应原文。
- 栏目名称。
- 当前书籍定位。

### 4.3 输出

```text
01-样文反推/反推记录/*.md
07-方法库/*.md
```

### 4.4 必须拆出

1. 标题机制。
2. 开头钩子。
3. 段落功能。
4. 痛点入口。
5. 论证结构。
6. 爽点句类型。
7. 原文依据使用方式。
8. 现实迁移方式。
9. 结尾套路。
10. 可入库方法。
11. 禁止入库项。

### 4.5 禁止

- 不得只写“这篇文章写得很好”。
- 不得只做摘要。
- 不得照搬竞品表达。
- 不得把竞品栏目当成所有书通用栏目。

## 5. chapter-extractor

### 5.1 核心职责

拆原文，提取文章生产所需的事实和依据。

### 5.2 输入

- 原文。
- 书籍定位。
- 栏目目标。

### 5.3 输出

```text
02-原文拆解/*.md
```

### 5.4 必须拆出

1. 写作背景。
2. 核心问题。
3. 关键事实。
4. 作者判断。
5. 论证步骤。
6. 关键概念。
7. 可引用原文。
8. 可迁移点。
9. 不能过度发挥的边界。

### 5.5 禁止

- 不得写成读后感。
- 不得现代化过度解释。
- 不得忽略原文证据。

## 6. story-long-analyze

### 6.1 核心职责

从原文拆解中提炼模型。

### 6.2 输入

- 原文拆解。
- 栏目协议。
- 样文反推结果。

### 6.3 输出

```text
03-模型提炼/*.md
```

### 6.4 可提炼模型

- 问题识别模型。
- 局势判断模型。
- 敌友识别模型。
- 决策推理模型。
- 行动路径模型。
- 组织协同模型。
- 认知纠偏模型。
- 现实迁移模型。

### 6.5 禁止

- 不得把模型写成空泛口号。
- 不得脱离原文强行总结。
- 不得为了整齐牺牲准确性。

## 7. story-long-analyze

### 7.1 核心职责

把书中的模型映射到现代用户处境。

### 7.2 输入

- 模型提炼。
- 目标用户。
- 栏目协议。

### 7.3 输出

```text
03-模型提炼/现实迁移/*.md
```

### 7.4 迁移要求

现实迁移必须满足：

1. 用户处境具体。
2. 与原文模型同构。
3. 不把历史问题硬套现代问题。
4. 能产生理解、判断或行动启发。

### 7.5 禁止

- 不得写成鸡汤。
- 不得替用户做现实决策。
- 不得把复杂现实简化成一句口号。

## 8. narrative-writer

### 8.1 核心职责

写用户可见栏目文章。

### 8.2 输入

- 书籍定位。
- 栏目协议。
- 样文反推。
- 原文拆解。
- 模型提炼。
- 现实迁移。

### 8.3 输出

```text
05-文章生产/成稿/*.md
```

### 8.4 写作要求

文章必须同时具备：

1. 用户问题入口。
2. 清楚的原文依据。
3. 能跟上的推理链。
4. 有力量的表达。
5. 不越界的现实迁移。

### 8.5 禁止

- 不得缺材料硬写。
- 不得把原文摘要当文章。
- 不得把栏目边界混在一起。

## 9. style-polisher

### 9.1 核心职责

提升文章表达质量。

### 9.2 可处理

- 标题更锋利。
- 开头更有抓力。
- 段落更顺。
- 爽点句更集中。
- 结尾更有余味。

### 9.3 禁止

- 不得改掉原文依据。
- 不得改掉核心判断。
- 不得为了爽感制造虚假结论。

## 10. consistency-checker

### 10.1 核心职责

审查文章是否可以进入下一步。

### 10.2 审查输出

```text
06-质量审查/*.md
```

### 10.3 审查等级

建议使用四级：

```text
S1 致命问题：不能发布，必须重写
S2 结构问题：需要大改
S3 表达问题：局部修改
S4 可优化项：不阻断
```

### 10.4 审查维度

1. 定位是否一致。
2. 栏目是否混乱。
3. 原文依据是否足够。
4. 推理链是否断裂。
5. 现实迁移是否过度。
6. 文章是否好读。
7. 是否有竞品照搬风险。
8. 是否有空话和鸡汤。

## 11. project-packager

### 11.1 核心职责

整理已经通过审查的项目资产。

### 11.2 输出

```text
08-项目打包/
```

### 11.3 第一阶段限制

第一阶段只做文件整理和交付说明，不做复杂产品化包装。

## 12. agent 协作顺序

标准顺序：

```text
story-architect
-> story-deslop
-> chapter-extractor
-> story-long-analyze
-> story-long-analyze
-> narrative-writer
-> style-polisher
-> consistency-checker
-> project-packager
```

其中：

- style-polisher 可选。
- project-packager 在第一阶段靠后。
- consistency-checker 必须在批量生产前执行。

## 13. 毛选样板中的默认分工

《毛选》第一阶段重点使用：

1. story-deslop：反推竞品底层解码篇和认知破局篇。
2. chapter-extractor：拆原文篇章。
3. story-long-analyze：提炼历史处境、误判、推理链、破局点。
4. story-long-analyze：把认知模型映射到现代痛点。
5. narrative-writer：写底层解码篇和认知破局篇。
6. consistency-checker：对标竞品审查。

## 14. 核心原则

agent 分工服务质量，不服务形式。

如果一个 agent 的输出不能帮助下一步写出更好的文章，就应该删掉或合并。
