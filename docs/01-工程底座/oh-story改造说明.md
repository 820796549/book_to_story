# oh-story 改造说明

## 1. 改造目标

oh-story 的价值不是“会写小说”，而是把垂直写作任务做成了工程化工作流。

本项目要借用它的工程方法，把小说生产底座改造成：

```text
拆有用书写文章生产系统
```

改造目标不是复刻小说项目，而是保留工程骨架，替换内容生产逻辑。

## 2. 保留什么

oh-story 中值得保留的是工程组织方式。

### 2.1 skill 路由

保留一个主入口 skill，用来判断用户当前要做什么。

拆书系统中对应为：

```text
$story
```

它负责把用户请求路由到：

- 项目初始化。
- 书籍定位。
- 样文反推。
- 原文拆解。
- 栏目写作。
- 质量审查。
- 项目打包。

### 2.2 专业 skill 分工

保留多个专业 skill，而不是让一个提示词承担所有任务。

拆书系统中对应为：

```text
$story-setup
$story-long-analyze
$story-long-analyze
$story-long-analyze
$story-long-analyze
$story-long-analyze
$story-long-write
$story-deslop
$story-review
$story-package
```

每个 skill 只处理一个明确阶段。

### 2.3 agent 职责边界

保留 agent 分工思想。

拆书系统不需要小说里的角色设计、伏笔追踪、章节写手，但需要自己的专业角色。

建议拆成：

- story-architect：书籍定位、用户定位、栏目体系。
- story-deslop：竞品样文反推、写法库沉淀。
- chapter-extractor：原文拆解、证据提取、篇章结构。
- story-long-analyze：模型提炼、推理链整理。
- story-long-analyze：现实迁移、用户场景映射。
- narrative-writer：栏目文章成稿。
- consistency-checker：质量审查、跑偏检测、修改建议。

### 2.4 hooks 守卫

保留 hooks 的“流程守卫”作用。

拆书系统中，hooks 重点防止：

- 未做书籍定位就写栏目。
- 未做样文反推就要求仿写。
- 未做原文拆解就生成正文。
- 栏目协议缺失时批量生产。
- 审查未通过就进入打包。

### 2.5 references 按需加载

保留 references 的方法库机制。

拆书系统中的 references 应该存放：

- 样文反推方法。
- 标题公式。
- 开头钩子。
- 痛点入口。
- 推理链模板。
- 爽点句类型。
- 现实迁移方法。
- 结尾套路。
- 禁止写法。
- 各类书籍栏目案例。

这些方法不应该一次性塞进上下文，而是由 skill 按需读取。

## 3. 替换什么

oh-story 的内容逻辑必须替换。

### 3.1 从小说项目替换为书籍项目

小说项目关注：

```text
题材
人设
世界观
剧情线
伏笔
章节
读者爽点
```

拆书项目关注：

```text
书籍价值
目标用户
栏目体系
原文依据
模型提取
现实迁移
文章表达
质量审查
```

### 3.2 从章节正文替换为栏目文章

小说系统写的是章节。

拆书系统写的是栏目文章。

栏目文章必须同时满足：

- 有明确用户问题。
- 有原文依据。
- 有推理链。
- 有现实迁移。
- 有文章读感。

### 3.3 从故事一致性替换为原文一致性

小说系统要检查设定冲突、角色状态、伏笔回收。

拆书系统要检查：

- 是否脱离原文。
- 是否误读原文。
- 是否把竞品表达当成自己的观点。
- 是否栏目边界混乱。
- 是否现代迁移过度。

### 3.4 从剧情爽点替换为认知爽点

小说的爽点来自冲突、反转、升级、打脸。

拆书文章的爽点来自：

- 一句话点破困境。
- 把混乱现实解释清楚。
- 把读者原本的误判翻过来。
- 把历史处境和当代处境打通。
- 给出可迁移的判断模型。

## 4. 使用方式目标

最终使用方式要接近 oh-story。

用户应该可以这样用：

```text
$story-setup
```

初始化一个拆书写作项目。

```text
$story-long-analyze
```

完成书籍定位、用户定位、栏目体系。

```text
$story-long-analyze
```

反推竞品或样文写法。

```text
$story-long-analyze
```

拆解原文篇章。

```text
$story-long-write
```

写栏目文章。

```text
$story-review
```

审查文章质量。

主入口：

```text
$story
```

负责根据用户意图自动路由。

## 5. 第一阶段改造范围

第一阶段只改到能支撑《毛选》样板生产。

必须完成：

1. 项目总纲。
2. skill 路由设计。
3. agent 职责设计。
4. hook 守卫设计。
5. 竞品反推标准。
6. 毛选栏目协议。
7. 单篇生产流程。
8. 质量审查标准。

暂不完成：

1. 多书并行调度。
2. 可视化界面。
3. 发布系统。
4. 复杂知识库检索。
5. 商业交付包装。

## 6. 改造验收标准

改造是否成功，看五件事：

1. 用户能像用 oh-story 一样调用拆书 skill。
2. 系统能阻止 AI 跳过定位、反推、原文拆解直接写正文。
3. 每个 agent 或 skill 的职责边界清楚。
4. 毛选样板能稳定写出底层解码篇和认知破局篇。
5. 产出的文章质量能通过审查，而不是只看生成速度。

## 7. 当前结论

本项目不是重新发明一套工程系统。

正确做法是：

```text
借 oh-story 的工程骨架，
替换小说生产逻辑，
装入拆书写文章流程，
先用毛选样板验证质量。
```
